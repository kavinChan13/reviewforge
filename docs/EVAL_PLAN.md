## 0. 为什么评测是质量体系的核心

代码审查 Agent 的价值不在"能跑"，而在"**报得准、漏得少、噪声小**"。一个**带数字**的评测体系把项目从"demo"变成可验收、可回归的工业产物：例如用基准集与消融实验呈现严重缺陷召回率、误报率的变化，以及相对 clang-tidy 单独使用在语义类缺陷上的增量价值。

本文定义如何得到这些数字。

## 1. 评测目标与指标

### 1.1 核心指标
| 指标 | 定义 | 目标 |
|---|---|---|
| **严重缺陷召回率 Recall** | 已知严重 bug 中被检出的比例 | ≥ 0.7 |
| **精确率 Precision** | 输出发现中"真问题"的比例 | ≥ 0.7 |
| **误报率 FPR** | 无效/噪声发现占输出比例 | ≤ 0.3 |
| **F1** | precision/recall 调和均值 | 跟踪趋势 |
| **定位准确率** | 发现行号落在缺陷真实范围内 | ≥ 0.8 |

### 1.2 工业体验指标
| 指标 | 定义 |
|---|---|
| 每改动平均发现数 | 控制噪声 |
| 审查时延 / token 成本 | 每 PR P50/P95 |
| 稳定性 | 同输入多次运行发现集的 Jaccard 一致性 |

### 1.3 评判方式（X12）
- **有标注样本**：与 ground-truth 缺陷位置/类别自动比对。
- **无标注/开放发现**：**LLM-as-Judge**（独立强模型）按 rubric 判定每条发现是否"真实、相关、可操作"，并人工抽检校准 judge。

## 2. 基准集来源（三路并行）

> 用户已确认**有真实历史缺陷种子可用** → 作为**高质量主标集**；合成注入与负样本作为规模化补充。

### 2.1 真实历史缺陷（主标集，最有说服力）
- 用户提供的（脱敏）**已知有问题的历史改动**：把"含 bug 的版本/diff"与"对应修复"配对，由修复反推缺陷位置/类别 → ground truth。
- 公开补充：知名 C++ 项目"修 bug 的提交"——fix commit 的父提交作为含 bug 版本，fix 内容反推标注。
- 价值：真实分布、最贴近工业，消融对比的结论最可信。

### 2.2 合成 bug 注入（规模化、零依赖）
- 取无 bug 的真实 C++ 代码，规则注入已知类别缺陷，每个注入即一条 ground truth：
  - 并发：去锁 / 改锁序 / 共享变量改非原子
  - 内存：删 `delete` / double-free / 返回局部引用 / 关 RAII
  - 安全：整型溢出、未校验长度、不安全 API
  - 正确性：off-by-one、删错误处理、空指针
- 优点：标注精确、类别分布可控、可大量生成。

### 2.3 负样本（测误报）
- 一批**确实没问题的正常改动**（重构、文档、格式化）→ 任何"严重发现"计为误报。衡量噪声的关键集。

> 基准集存放 `benchmarks/`，每条：`{id, repo_or_patch, diff, ground_truth_findings[], label_source}`。真实种子建议放 `benchmarks/real/`，合成放 `benchmarks/synthetic/`，负样本放 `benchmarks/negative/`。

## 3. 消融实验（量化各设计对指标的影响）

| 配置 | 说明 | 验证的假设 |
|---|---|---|
| **Baseline-A** | clang-tidy 单独 | 传统静态分析基线 |
| **Baseline-B** | LLM 只看 diff（无 RAG、无静态分析） | 朴素 LLM 审查 |
| **+RAG** | LLM + 代码库上下文 | 上下文降误报、提相关性 |
| **+静态分析融合** | LLM + RAG + clang-tidy 信号 | 事实锚点降幻觉、提召回 |
| **+多维子 Agent** | 完整状态图编排 | 维度分工提升深度缺陷召回 |
| **+记忆/抑制** | 完整 + 长期记忆反馈闭环 | few-shot 提精确率、噪声下降 |

**实测结论（公开子集，10 个 case，可复现，见 [`benchmarks/results-public/`](../benchmarks/results-public/)）**：
- `+RAG / full` 相对 `B-llm-only` **recall 单调提升**（v2 40%→60%、v3 40%→50%），并保持 **100% 精确率 / 近 0 误报 / 100% 行号定位**；
- 单次 LLM 运行方差较大（实测 recall ±19pt），可靠对比需 `--runs N` 多次取均值并固定缓存策略，单次消融的小差异落在噪声内；
- 内部专有 C++ 历史 case（不可公开复现，仅定性参考）：调优后 Recall 87.5% / Precision 77.8% / F1 82.4% / FP 0.67/PR，相对未调优基线 F1 ×2.5、误报 ×11.5↓。

> 注：测试环境未安装 clang-tidy 时，`+static` 配置实际等价于 `+rag`；装上 compile DB 后该列预期有显著提升。

## 4. 评测流程与产物

```
benchmarks/* → eval/runner 跑各配置 → 收集 findings
  → 自动比对 ground truth（有标注）+ LLM-as-Judge（开放发现）
  → metrics 计算 Recall/Precision/FPR/F1/定位准确率/时延/成本
  → 生成报告：每配置一行对比表 + 消融曲线 + 误报/漏报样例
```
产物：
- `benchmarks/results/<timestamp>.json`（原始）
- `benchmarks/results/report.md`（对比表 + 消融结论 + 坏案例）
- CI 可选：指标回归门禁（掉超阈值告警）

---

## 5. 校准与诚信
- LLM-as-Judge 需**人工抽检校准**（随机抽 N 条核对一致性），报告注明 judge 模型与一致率，避免"自己给自己打分"偏差（X12）。
- 报告同时给出**坏案例（误报/漏报样例）**，便于判断结论可信度并明确后续改进优先级。

## 7. 已定决策（原"待确认"，现已落地）

1. **真实种子提供格式** —— 采用**方案 A**：给 `仓库 + 修复提交 hash`，由 `scripts/seed-from-commit.ts` 自动从 git 抽 diff 并反推 ground truth。
2. **仓库可访问性** —— **两类都用**：公开仓库（spdlog C++、tidwall/gjson Go）进公开基准集；内部专有 C++ 仅本地脱敏运行、不纳入公开仓库。
3. **LLM-as-Judge 模型** —— 已实现，**可通过 `JUDGE_MODEL` 环境变量**指向更强 / 不同家的模型（降同源偏差）；默认回退到审查模型。
4. **CI 回归门禁** —— 门禁**机制已实现**（`rf eval --baseline <report.json>`，回归即非零退出）；目前**尚未接入自身自动化 CI**（`.github/workflows/ci.yml` 仅跑 typecheck + 单测），自动化接入留作 roadmap。
